# 分析結果が「なんか違う」と感じる理由 ── 離散分類の限界と、次の設計

> ステータス：草稿。形になったタイミングでnote公開予定。

---

thought-analyzerの分析結果を見て、「自分っぽくない」と感じたことはないだろうか。

軸の値は合っているのに、全体像として何かが欠けている感覚。その正体は、測定設計の構造的なトレードオフにある。問題を正直に書き、現在考えている解決策を共有する。

---

## `pivot` と「55%pivot」は、同じ値になる

現在のthought-analyzerは、各軸に「支配的な値1つ」を割り当てる設計だ。

たとえば `problem_style: pivot`。しかしこれは、その人が会話の中で80%pivotだった場合も、51%pivotだった場合も、同じ値になる。境界付近にいる人の「揺れ」が消える。

さらに問題なのは、**軸間の相互作用が失われること**だ。`need_for_cognition: high` かつ `problem_style: suspend` の組み合わせは、「深く考えるが結論を急がない」という一貫した人格を示す可能性がある。しかし現在の出力では、この関係性は読み取れない。

測定理論の言葉を借りれば、離散分類は名義尺度（nominal scale）であり、連続値は比率尺度（ratio scale）に近い。情報量は後者の方が圧倒的に多い。離散化は意図的なトレードオフだったが、その代償は小さくない。

---

## 短期の解決策：数値で測れないものを、言葉で補う

最もコストが低く、効果が高い改善がナラティブ層の追加だ。

軸分析（構造）の後に、**同じClaudeが会話全体を振り返り、軸では表れない傾向を自然言語で記述するフェーズ**を加える。これを `holistic_profile` フィールドと呼んでいる。

```
Step 1: 構造分析（軸 → JSON）
Step 2: 会話全体を振り返り、holistic_profileを記述
```

ここに重要な洞察がある。**そのユーザーのことを最もよく知っているのは、その場にいたClaudeだ。** 軸分析は「分類のための抽象化」を強いるが、会話をしていたClaudeは分類に収まらない情報——言葉の選び方、話題の飛び方、前提の置き方——を保持している。その情報を引き出すのがStep 2の役割だ。

定量（軸）＋定性（ナラティブ）の二層構造で、どちらか一方では捉えられないものを補完し合う。DBスキーマの変更は不要で、skill.mdの修正だけで実現できる。このフェーズはすでに実装済みだ。

---

## 中長期の課題：値を連続化する

より根本的な改善は、離散値から連続スコアへの移行だ。

```json
// 現在（v2.0）
"problem_style": "pivot"

// 連続化後（v3.0構想）
"problem_style": {
  "primary": "pivot",
  "secondary": "fix",
  "consistency": 0.72
}
```

`consistency` は「どれだけ一貫した傾向を示しているか」を0〜1で表す。低いほど「揺れている」ことを示し、境界付近のユーザーを正確に記述できるようになる。

ただし、この改善が価値を持つのはサンプルが蓄積されてからだ。n数が少ない今は、連続値の分布を見ても意味がない。**n=100を超えたタイミングで連続化に移行するのが自然な流れ**だと考えている。DBスキーマの変更と後方互換性の処理が主なコストになる。

---

## 将来の構想：分析を対話にする

さらに先の構想として、軸分析の結果をClaudeがユーザーと「対話」しながら補正していくアプローチがある。

分析結果をもとにClaudeがユーザーへ質問を投げかけ、回答を踏まえて分析を更新する。「自己評価」と「Claude評価」の差分を可視化することで、自分では気づいていない傾向が浮かび上がる可能性がある。

測定精度の向上というより、**分析自体を対話的なプロセスにする**発想の転換だ。実装コストは高いが、ユーザーエクスペリエンスとしての可能性は大きい。

## 今の優先順位

```
┌──────────────────┬──────────────────────────┬──────┬──────┐
│ フェーズ          │ 施策                      │ 工数 │ 効果 │
├──────────────────┼──────────────────────────┼──────┼──────┤
│ 実装済み          │ holistic_profile（ナラティブ）│ 小   │ 中〜大│
│ n数100以上        │ 軸の連続化（v3.0）         │ 中   │ 大   │
│ 将来              │ 対話型最終評価             │ 大   │ 未知数│
└──────────────────┴──────────────────────────┴──────┴──────┘
```

現時点ではナラティブ層が最良のコスパだ。v3.0の連続化はサンプルが積み上がってから判断する。

n数が増えるほど、測定の設計選択が問われるようになる。その前に構造を整えておくことが、今の課題だ。
